# 小说结构修订助手

对小说初稿进行深度结构诊断，发现问题并制定修订方案。

## 核心定位

```
novel-revision（本技能）= 诊断与规划
├── 发现问题：系统性诊断结构问题
├── 分析问题：评估影响和优先级
├── 制定方案：提供可执行的修订计划
└── 讨论确认：与用户确认修订方向

novel-writing = 执行与控制
├── 调用本技能进行诊断
├── 执行具体修订操作
└── 追踪修订进度
```

## 触发前提

- 用户已完成初稿（`05-小说初稿/`）
- 用户想要诊断或修订作品结构

## 交互式诊断流程

### 第一步：确认诊断范围（使用 AskUserQuestion 工具）

```
问题：请选择诊断范围？
选项：
- 全文诊断：系统分析整部作品
- 单元诊断：分析某个叙事单元（如第一幕）
- 重点诊断：针对特定问题（如情节漏洞、人物弧线）
- 章节诊断：分析单个章节
```

### 第二步：执行诊断

根据范围执行诊断，输出问题清单。

### 第三步：讨论确认（使用 AskUserQuestion 工具）

**关键步骤** - 与用户讨论问题清单：

```
问题：您认同这些问题清单吗？
选项：
- 认同，继续制定修订计划
- 需要调整某些问题
- 有补充问题
- 重新诊断
```

### 第四步：制定修订计划

确认问题后，制定可执行的修订计划：

```markdown
# 修订计划

## 优先级排序
1. [问题A] → 修订方案 → 预计工作量
2. [问题B] → 修订方案 → 预计工作量
3. ...

## 修订策略
- [ ] 需要回溯修改前文铺垫
- [ ] 需要调整人物设定
- [ ] 需要补充新场景
- [ ] 需要删减冗余内容

## 执行建议
- 建议使用 novel-writing 执行修订
- 预计总工作量：[估算]
- 建议分 [N] 次会话完成
```

### 第五步：确认下一步

```
修订计划已制定。您希望：

[1] 我来执行修订 - 切换到 novel-writing 模式
[2] 先保存计划，稍后执行
[3] 深入讨论某个问题的解决方案
[4] 调整修订计划
```

## 诊断维度

### 一、情节诊断

```markdown
## 情节问题检查

### 逻辑漏洞（致命）
- 时间线是否一致？
- 因果关系是否完整？
- 角色是否"知道不该知道的事"？
- 事件结果是否与设定矛盾？

### 情节断点（严重）
- 每条线索是否有交代？
- 伏笔是否有呼应？
- 配角去向是否交代？
- 支线是否完整闭合？

### 动力问题（严重）
- 主角动机是否清晰？
- 反派行动是否有逻辑？
- 关键转折是否有铺垫？
- 高潮是否水到渠成？
```

### 二、结构诊断

```markdown
## 结构问题检查

### 开篇问题（致命）
- 前500字是否吸引人？
- 前3章是否建立悬念？
- 主角是否足够快出场？

### 节奏问题（严重）
- 是否有章节拖沓？
- 中段是否塌陷？
- 高潮是否足够紧张？

### 结构问题（严重）
- 三幕比例是否合适？
- 转折是否生硬？
- 结局是否仓促？
```

### 三、人物诊断

```markdown
## 人物问题检查

### 主角弧线（致命）
- 开场状态是否清晰？
- 成长是否有铺垫？
- 转变是否有说服力？
- 结局是否完成弧线？

### 人物一致性（严重）
- 行为是否符合性格？
- 对话是否符合人设？
- 前后行为是否矛盾？

### 人物功能（轻微）
- 是否有冗余人物？
- 每个人物是否有存在意义？
```

### 四、主题诊断

```markdown
## 主题问题检查

### 主题呈现（严重）
- 核心主题是否明确？
- 是否通过情节展现？
- 是否避免说教？

### 主题深度（轻微）
- 是否有足够的复杂性？
- 是否有不同视角？
```

## 问题优先级标准

| 级别 | 定义 | 示例 | 处理建议 |
|------|------|------|----------|
| 🔴 致命 | 破坏故事基本逻辑 | 情节漏洞、人物崩坏、主题偏离 | 必须修改 |
| 🟡 严重 | 影响阅读体验 | 节奏拖沓、弧线断裂、结局仓促 | 建议修改 |
| 🟢 轻微 | 可优化但不影响大局 | 对话不够自然、细节可精简 | 可选修改 |

## 常见问题解决方案库

### 情节漏洞

```
问题：主角突然知道不该知道的信息

解决方案：
1. 增加获取信息的场景（推荐）
2. 改变信息来源
3. 调整主角的知情范围

示例修改：
- 在第3章增加主角偷听对话的场景
- 或：让配角在适当时机告知
```

### 节奏拖沓

```
问题：中间章节进展缓慢

解决方案：
1. 删减无推进作用的场景（推荐）
2. 合并相似场景
3. 增加新的冲突元素

判断标准：
- 该场景是否推动情节？
- 是否展现人物？
- 是否提供关键信息？
三者皆无 → 可删除
```

### 人物崩坏

```
问题：人物行为与设定矛盾

解决方案：
1. 增加行为转变的铺垫（推荐）
2. 增加内心挣扎的描写
3. 调整行为本身
4. 通过其他人物反应合理化

示例修改：
- 在转变前增加2-3处心理描写
- 通过配角的质疑来呼应读者的疑惑
```

### 结局仓促

```
问题：结局来得太快太突然

解决方案：
1. 增加高潮前的铺垫章节（推荐）
2. 在前文埋下更多伏笔
3. 减缓高潮前的节奏
4. 增加结局后的收束场景

经验法则：
- 结局篇幅应占全文10-15%
- 高潮前应有"暴风雨前的宁静"
```

## 文件管理规范

### ⚠️ 重要原则：永不覆盖原始文件

```
❌ 错误做法：直接修改 05-小说初稿/ 中的文件
✅ 正确做法：将修订稿输出到 06-结构修订/修订稿/

初稿是创作的历史记录，必须保留！
修订稿是独立的版本，便于对比和回溯。
```

### 目录结构

```
06-结构修订/
├── 诊断报告.md           # 问题清单与分析
├── 修订计划.md           # 可执行的修订方案
├── 修订记录.md           # 执行过程中的记录
├── 修订稿/               # ⭐ 修订后的章节（独立文件）
│   ├── 第1章.md
│   ├── 第2章.md
│   └── ...
└── 版本对比/             # 可选：原文与修订对比
    ├── 第1章-对比.md
    └── ...
```

### 修订稿命名规范

```
修订稿文件名 = 章节名.md（与初稿相同）

示例：
初稿位置：05-小说初稿/章节/第1章.md
修订稿位置：06-结构修订/修订稿/第1章.md
```

### 版本对比格式

```markdown
# 第X章 版本对比

## 修订摘要
- 修订类型：[情节补充/人物调整/删减优化]
- 修订原因：[问题说明]
- 修订效果：[预期改善]

## 对比

### 原文（初稿）
> [原文内容]

### 修订后
> [新内容]

### 修订说明
[为什么这样改，解决了什么问题]
```

## 与其他技能的协作

```
用户请求 → 判断触发哪个技能

"修订小说" → novel-revision（诊断）
                ↓ 确认计划后
            → novel-writing（执行修订）

"继续写作" → novel-writing（写作控制台）
                ↓ 需要诊断时
            → novel-revision（诊断）

"润色文字" → novel-polish（文字润色）
```
